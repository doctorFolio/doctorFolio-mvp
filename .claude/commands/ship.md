현재 브랜치의 변경사항을 리뷰하고, 블로커가 있으면 Codex에게 수정을 위임한 뒤 PR을 생성합니다.
최대 3회 반복합니다.

## 실행 루프

### Step 1: 검증
`bun run verify` 실행. 타입/린트/테스트 오류가 있으면 해당 오류 전체를 기록합니다.

### Step 2: 리뷰
`git diff main...HEAD`로 변경 내용을 확인하고 4관점으로 검토합니다.

#### 🔒 보안
- API 라우트 입력 검증 누락
- 환경변수 클라이언트 번들 노출 위험
- Rate limit 우회 가능성
- 금융 데이터 무결성 (계산 결과 변조 경로)

#### 🏗️ 유지보수성
- Tidy First 위반: structural + behavioral 혼재
- 파일 200줄 / 함수 50줄 초과
- props drilling 3단계 이상
- 중복 티커 자동 제거 금지 규칙 위반

#### ⚡ 성능
- 불필요한 리렌더 (인라인 객체/함수 props)
- API 라우트 동기 블로킹
- Claude API 호출 과다

#### 🧪 테스트
- 새 순수 로직에 테스트 없음
- 엣지 케이스 누락 (빈 포트폴리오, 0원, 100% 단일 종목)

### Step 3: 판정

리뷰 결과를 다음 형식으로 출력합니다:

```
## 리뷰 결과 (N/3회차)

| 관점 | 상태 | 블로커 |
|------|------|--------|
| 🔒 보안 | ✅/❌ | ... |
| 🏗️ 유지보수성 | ✅/❌ | ... |
| ⚡ 성능 | ✅/❌ | ... |
| 🧪 테스트 | ✅/❌ | ... |

**블로커 목록:**
- [파일:라인] 문제 설명
```

**블로커 없으면** → Step 4로 이동
**블로커 있고 3회 미만이면** → Step 3a로 이동
**블로커 있고 3회 소진했으면** → 중단하고 블로커 목록 보고

### Step 3a: Codex에 수정 위임

블로커 목록을 다음 형식으로 정리해서 Codex에게 전달합니다:

```bash
codex --full-auto "다음 문제들을 수정해줘. 수정 후 bun run verify가 통과해야 함.

[블로커 목록을 구체적으로 기술 — 파일명, 라인, 문제, 수정 방향]

주의:
- structural 변경과 behavioral 변경은 별도 커밋
- any 타입 사용 금지
- 테스트가 있는 로직은 테스트도 업데이트"
```

Codex 실행이 끝나면 Step 1로 돌아갑니다.

### Step 4: PR 생성

```bash
gh pr create --fill
```

PR 생성 후 다음을 보고합니다:
- 총 반복 횟수
- 최종 리뷰 결과 요약
- PR URL
