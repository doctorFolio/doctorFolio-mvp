현재 브랜치의 변경사항을 리뷰하고 PR을 생성합니다.

## 실행 순서

### 1. 검증
`bun run verify` 실행. 실패하면 즉시 중단하고 오류를 보고합니다.

### 2. Diff 파악
`git diff main...HEAD` 로 변경된 파일과 내용을 확인합니다.

### 3. 4관점 순차 리뷰

각 관점을 **독립적으로** 검토합니다. 이전 관점의 판단이 다음에 영향을 주지 않도록 합니다.

#### 🔒 보안 (Security)
- API 라우트 입력 검증 누락 여부
- 환경변수 노출 위험 (클라이언트 번들에 포함 가능성)
- Rate limit 우회 가능성 (`middleware.ts` 확인)
- XSS / injection 가능성 (사용자 입력을 직접 렌더링하는 경우)
- 금융 데이터 무결성: 계산 결과가 변조될 수 있는 경로 존재 여부

#### 🏗️ 유지보수성 (Maintainability)
- Tidy First 위반: structural + behavioral 변경이 한 커밋에 섞였는지
- 파일 200줄 / 함수 50줄 초과 여부
- props drilling 3단계 이상 여부
- 같은 로직이 3곳 이상 중복되는지
- 중복 티커 자동 제거 금지 규칙 위반 여부 (AGENTS.md 도메인 규칙)

#### ⚡ 성능 (Performance)
- 불필요한 리렌더 유발 가능성 (인라인 객체/함수를 props로 전달하는 경우)
- API 라우트에서 동기 블로킹 작업 여부
- Claude API 호출이 필요 이상으로 많이 발생하는 경로 존재 여부
- 이미지 처리 메모리 사용 (OCR 라우트)

#### 🧪 테스트 (Test Coverage)
- 새로운 순수 로직 (`engine.ts`, `sectors.ts` 등)에 대한 테스트 존재 여부
- 엣지 케이스 누락: 빈 포트폴리오, 100% 단일 종목, 0원 자산
- API 라우트는 단위 테스트 대상이 아님 (통합 테스트 범위)

### 4. 결과 출력

```
## /ship 리뷰 결과

### 🔒 보안
| 체크포인트 | 상태 | 상세 |
|-----------|------|------|
| 입력 검증 | ✅ | |
| 환경변수 노출 | ✅ | |
| Rate limit | ✅ | |

### 🏗️ 유지보수성
| 체크포인트 | 상태 | 상세 |
|-----------|------|------|
| Tidy First | ✅ | |
| 파일/함수 크기 | ✅ | |
| 중복 코드 | ✅ | |

### ⚡ 성능
| 체크포인트 | 상태 | 상세 |
|-----------|------|------|
| 리렌더 위험 | ✅ | |
| API 블로킹 | ✅ | |

### 🧪 테스트
| 체크포인트 | 상태 | 상세 |
|-----------|------|------|
| 새 로직 커버리지 | ✅ | |
| 엣지 케이스 | ✅ | |

---
**블로커:** 없음 / [목록]
**판정:** PASS → PR 생성 / FAIL → PR 생성 중단
```

### 5. 판정
- **PASS (블로커 없음):** `gh pr create --fill` 실행
- **FAIL (블로커 있음):** 블로커 목록만 출력하고 중단. PR 생성하지 않음.

> ⚠️ 경고(warning)는 블로커가 아닙니다. 개선 제안은 PR 생성 후 코멘트로 남깁니다.
